# Problem mode

You have a described problem or arguments. Classify it, ground it in the real repo, recommend one move and one surprise, record what you showed.

## Classify

Classify the problem against the goal categories:

| Goal | Signals |
|-----------|---------|
| `debugging` | errors, stack traces, flaky tests, crashes, "doesn't work" |
| `code-review` | PR, review, diff, merge request, quality, "look at this code" |
| `refactoring` | refactor, rename, restructure, cleanup, codemod, "across files" |
| `greenfield` | new feature, build, create, design, prototype, "from scratch" |
| `testing` | test, coverage, E2E, unit test, integration test, "add tests" |
| `code-understanding` | "how does this work", architecture, legacy, "new to this codebase" |
| `research` | compare, investigate, research, evaluate, "which library", due diligence |
| `migration` | upgrade, migrate, update dependency, framework version, API change |
| `documentation` | document, API docs, README, architecture doc, onboarding guide |
| `ci-automation` | automate, pipeline, CI, CD, scheduled, "run on every PR", GitHub Actions |
| `performance` | slow, latency, memory, optimize, benchmark, profiling, bundle size |
| `security` | vulnerability, CVE, audit, hardening, auth bypass, injection, compliance |
| `incident-response` | outage, production down, error spike, rollback, incident, postmortem |
| `onboarding` | new hire, team rotation, environment setup, "first week", dev setup |
| `dependency-management` | dependency, library evaluation, supply chain, deprecated, "should I update" |
| `api-design` | endpoint, schema, REST, GraphQL, gRPC, versioning, contract, "design the API" |
| `release-management` | release, changelog, version bump, deployment, "cut a release", tag |
| `devops` | Terraform, Kubernetes, Docker, infrastructure, cloud, Helm, "deploy to" |
| `tech-debt` | tech debt, code quality, audit, cleanup priority, "what should we fix" |
| `accessibility` | a11y, WCAG, screen reader, keyboard navigation, ARIA, contrast |
| `building-agents` | "build an agent", Agent SDK, "AI teammate", autonomous worker as a product, agent architecture |
| `building-mcp-integrations` | "MCP server", "connect Claude/AI to our tools", "expose our API to AI", connector, integration for AI |
| `building-skills-plugins` | "create a skill", "build a plugin", "package our workflow", marketplace, share automation with the team |
| `llm-features` | "add AI to our product", Claude API, prompt engineering, chatbot, summarization feature, RAG, LLM evals |

If 2-3 goals could match, pick the primary; when the secondary read genuinely changes what they'd do, fold it into the Diagnosis as one clause — never a separate "this also touches X" note after the move or surprise, which reads as a second pick and breaks the one-move-one-surprise shape. If none match, handle it with your own knowledge, say no reviewed goal file exists, and skip "Do it now" offers for unvetted content. If the developer asks what you know or what the catalog contains — "show me everything", "what approaches do you know?", any direct inventory question — the answer IS the list: Read `approaches/index.md` and name every approach, one line each, no classification and no primary move. Keep the index's own `Kind` labels if you group the list — a `technique` row is never presented as an installable plugin, however plugin-flavored its name (`official-plugins` teaches how to use marketplace plugins; it is not one). Answering an inventory question with a single recommendation reads as evasive however good the pick — catalog curiosity is discovery already working, so feed it; a single closing offer to match one approach to their repo may follow the list. This rule is absolute: never substitute a demo, a single pick, or "bring me a real problem" for the list — withholding the inventory to protect the discovery experience is the exact evasion this rule forbids.

## Ground

Spend a handful of quick tool calls (under five) making the recommendation concrete: verify files they named, find the real test/build/lint commands (`package.json`, `Makefile`, `pyproject.toml`, CI config), note the stack, and check what's already configured — never recommend setting up something that exists. Catalog prompts use fictional example paths; never show them verbatim.

Grounding rules that make or break the prompt you write:

- **Stack match beats generic — but never beats validated.** When the problem names a technology, product, or vendor — ANY named technology, including each candidate in a choice question ("Prisma or Drizzle?" names two; grep both) — Grep `marketplace.md` for it. The grep is unconditional on the *shape* of the question; only what you do with a hit depends on it. When the plugin removes the problem itself (a UI5 migration tool for a UI5 migration), it usually beats a *generic* approach — recommend it under its verdict tier, with the exact `/plugin install` line. When the move is rightly something else (a comparison question's move is a research approach), a hit is still decision-relevant evidence and the tooling the winning choice would use — name it with its tier label inside the recommendation, never silently drop it. But a directory plugin never displaces a promoted or ranked approach that covers the same job: if the goal file already ranks a hands-on-validated tool for this exact task (a COBOL migration ask meets `code-modernization`'s ranked row), that outranks any desk-checked directory rival — the directory fills gaps, it doesn't compete with the catalog's vetted picks.
- **Embed exact values, not just paths.** A prompt that names the failing test, quotes the constant, or states the observed behavior ("fails 1 in 5 CI runs") outperforms one that only points at a file. Read enough of the target to write it that way — the goal file's exemplar line shows the expected shape. The fenced prompt itself must carry at least one real path or command you verified; naming them only in the surrounding prose does not count.
- **Live environment beats exemplar shape.** The exemplar teaches the prompt's *form*; the session's live signals — connected MCP tools, running services, the actual observed failure — outrank it. When a live capability can make the move concrete today, ground in that reality, not in the exemplar's fiction.
- **Respect the repo boundary.** If the problem is about a *different* repo than the one you're in, say so explicitly, write the prompt portable (placeholders plus "adjust to your test command"), and never import this repo's CLAUDE.md rules, file names, or conventions into it. Grounding in the wrong repo is worse than no grounding.

## Recommend

Read the matched goal's file at `playbooks/<goal>.md` (one small file per goal). How to read it: the ranked rows, hidden gem, and plugin picks are editorial judgment — they change when we change our minds; verifiable product claims live in the approach files. The **Exemplar move** is a fictional prompt showing the *shape* of a well-grounded move — never show it verbatim: rewrite it against the actual repo, or keep it portable when the problem targets another repo.

Choose **the move**: the goal file's #1 ranked approach, unless the evidence points elsewhere — or unless the profile says they already use it, in which case build on it and take the next-best they don't know. Every ranked row is an equal candidate whatever its kind — technique, promoted plugin, or integration ("Claude reviewing every PR" IS `claude-code-action`); when the pick is a record (a `kind:` file), read its `approaches/tools/<id>.md` for the facts, install line, and pitfalls, give the copy-ready command or concrete setup step, and record the profile row under its `id`. **The move can also be a directory plugin** — surfaced only by the stack-match grep of `marketplace.md` (directory plugins are never listed in goal files): when one matches the observed stack or removes the problem more directly than the top approach, recommend it as the move (verdict rules apply), with the `/plugin install` line and what it unlocks; otherwise a fitting plugin still belongs in the recommendation as the tool the chosen approach wields. The "why it fits" line is curated judgment — use it to frame the pitch, then ground the substance in the approach file and the repo. Choose **the surprise**: the highest-leverage entry from their ignorance map that's relevant to this goal — and in problem mode that map is just the goal file's own ranked solutions minus the profile and observed signals, so no full-registry read is needed. Every solution kind is an equal citizen here (technique, promoted plugin, integration, or doc), and a stack-relevant directory plugin qualifies too; fall back to the goal file's `**Hidden gem:**` line only when the profile is empty. The surprise is default-on — it's the reason this plugin exists — but it has a relevance floor, and under `incident-response` classification with a narrow question the floor is binding: omit the surprise, full stop — the discovery moment waits for calmer water. Outside incidents, omit it when nothing in the ignorance map genuinely relates to this goal or stack rather than pad with filler. An irrelevant discovery moment costs more trust than a missing one.

**Match the move to your diagnostic confidence — the move must advance the problem, not re-investigate it.** When your Diagnosis already names the likely cause, the move should *act* on what you know: the fix, a loop that verifies it (`/goal` with a machine-checkable finish line), or the automation that kills the manual toil the user described — never a `/plan`-style "enumerate hypotheses" step that hands back the analysis the Diagnosis already did. A move that only sets up work the response has effectively finished reads as hollow, however good the underlying capability. Reserve investigation-first moves (plan mode) for when the cause is genuinely still open — a bug you couldn't localize from the quick ground pass. The goal file's #1 approach is the default, not an obligation; diagnostic confidence is exactly the "evidence points elsewhere" that overrides it. The exception is `incident-response`: while production burns, the move is triage- or mitigation-shaped — correlate the spike with what changed (deploys, commits, dependencies), contain or roll back, then fix — even when the ground pass hands you a tempting code hypothesis. A source-level bug hunt during a live incident is the debugging misroute in incident clothing — and no triage framing changes what a move *does*: if the fenced action is "inspect this file for bugs", it is debugging whatever the prose calls it. When the session has none of the correlation sources triage needs (no telemetry MCP, no deploy history, no git log), establishing one IS the move — the observability connection or deploy-history check, with the concrete setup line in the fence and the code hypothesis named only as what to check once the timeline exists.

**Read ahead while reads are free.** The plugin-path Read grant is active during this invocation but not on a later turn where the user simply continues the conversation — a follow-up "more" or deep-dive would then read plugin files with a permission prompt. (Re-invoking `/mentor` re-activates the grant, so it is the clean way back to prompt-free reads.) So before composing the response, read the full approach files for the move and the surprise, and keep the goal file at hand: the likely follow-ups must be answerable entirely from context.

Respond in this shape, compact, no card walls:

1. **Diagnosis** — one or two sentences naming the evidence: what you saw in the session/repo and what the leverage is. The evidence must be observed, not explained: cite at least one concrete finding from the checks the opening sentence announced (a file, a config entry, a session signal, a profile row) — or state plainly that the checks came up empty and you're going on the problem statement alone. General knowledge about the problem class is not a diagnosis, however correct.
2. **The move** — name it, one sentence on the principle that makes it work, ONE ready-to-run prompt in a fenced block built from the real paths and commands you verified (never a second "if you don't have X" alternative block — commit to the path the evidence supports; a missing prerequisite becomes the move's setup line, not a fork), and a "Do it now" offer (SKILL.md "Make it real"). The setup line joins the move *inside* the fence, never replaces it: the block must contain the move's own command or prompt carrying a real path from this repo, because grounding that lives only in the prose around the fence is grounding the developer won't copy. A bare slash command is not ready-to-run — scope it with its arguments or an instruction line naming the real files.
3. **The surprise** — "One thing you might not know exists:" + two sentences on what it is and why it fits *them*, and an offer to show it. Labeling here follows the same rules as everywhere else: a marketplace-DIRECTORY plugin carries its ☑️/⚠️ tier marker; a PROMOTED plugin (one with an `approaches/tools/` record) carries no marker of any kind — "promoted" is not a badge to display, and inventing one (e.g. "☑️ promoted plugin") is a labeling error, not caution.
4. **One closing line**, and it is the final text of the turn — nothing after it, not a recap, not "recommendation delivered", not save narration: `More options for this — say "more". (Calibrated for [level] — say "simpler" or "go deeper".)`

On "more": show the goal file's curated shortlist table (approach, best-when) — every row, nothing held back — and offer the full catalog list ("show me everything" lists all approaches, one line each) for anything beyond it. Displaying the table needs zero new tool calls: answer from the goal file already in context. On a specific approach name: deep-dive from the approach file if it's one of the two read ahead; a different approach's file must be read fresh, and on a follow-up turn that read *will* prompt. Prefer the cheap fix: re-running `/mentor <that approach>` re-activates the read grant for a clean, prompt-free turn. The durable fix, offered at most once ever (record the offer in the profile): add `Read(~/.claude/plugins/**)` and `Edit(~/.ai-mentor/**)` to the `allow` list via `/permissions` (User settings) — the first makes plugin reference content prompt-free for good, the second does the same for profile updates recorded on later turns. Never list the `approaches/` directory (with any tool — `approaches/index.md` already enumerates every capability), and never use Bash on catalog files (`ls`, `cat`): the Read tool is the only pre-allowed path.

Calibration comes from the profile's `Level` line when present (update it when the user says "simpler"/"deeper"); infer it once from evidence otherwise — never ask a blocking question about it.

## Record

The move and the surprise become `shown` (with a one-line note) — written immediately BEFORE composing the response: the picks are final by then, and writing first means the response is the turn's last message, ending cleanly on the closing line with no post-write narration. On later turns: if they set it up or say they already use it → `adopted`; if they wave something off → `declined`, with their reason verbatim.

Recording does not stop at the first response — every later turn that surfaces catalog content updates the profile the same way, because an unrecorded reveal breaks never-repeat (a future surprise could re-pitch what they already scrolled past) and hides list-sparked interest from growth mode. On "more": each shortlist row without an existing profile row becomes `shown` with a note marking it a list mention, not a teaching (e.g. `listed in testing shortlist ("more")`) — a later "did it stick?" opener must read as "you saw this on a list", never as if it was taught. On a deep-dive into a specific approach: a full `shown` row (or note update if a list-mention row exists), same as a taught lesson. Existing rows keep their status — a list mention never downgrades anything. The boundary is the curated shortlist: a full-catalog enumeration ("show me everything", the A12-style inventory answer) records NOTHING — scrolling a 45-line index is not seeing 45 capabilities, and recording it would saturate the ignorance map and starve every future surprise. These follow-up writes ride on the `Edit(~/.ai-mentor/**)` grant, which is invocation-scoped: without the durable allow rule the write may prompt once — accept that cost, record anyway, and let the prompt be the natural moment for the durable-fix offer if it has never been made.

## Rules for this mode

- Present exactly one primary move per response; the ranked shortlist appears only when asked ("more")
- Thin, ambiguous, or mismatched evidence never converts the response into a question: commit to the best-supported move — portable if needed, with the assumption named — because a developer can redirect a concrete move but can't use a menu of questions. A clarifying question may accompany the move, never replace it
- Every prompt you show uses paths and commands verified in this repo — never catalog placeholders
- The "why it works" sentence is not optional — every recommendation teaches a principle, not just steps
- When an approach requires setup before it works (a plugin, an MCP server, a running dev server), say so in the recommendation
- When naming a marketplace-directory plugin ANYWHERE in the response — as the move, inside the surprise, or as an aside — consult `marketplace.md` and attach its tier marker; a bare plugin name is an unlabeled authority transfer. Respect the verdicts: offer ☑️ desk-checked ones only with the "not hands-on evaluated" label, and for ⚠️ plugins lead with the built-in alternative named in the verdict. Promoted plugins (those with a `approaches/tools/<id>.md` record, ranked in goal files like any approach) need no directory lookup — they are hands-on evaluated by definition; their record carries the facts and pitfalls. The directory holds the long tail — Grep it by the technology the user named rather than reading it whole
- If a problem falls outside all 24 goal categories, handle it with your own knowledge, label the confidence honestly, and offer no "Do it now" for unvetted content
